ZMHSTUDIO

a one-human game studio, run like a big one

One person sets the creative direction. An AI lead engineer plans, builds, tests, and ships. This page explains the machine that makes that work.

Creative DirectorZack — the human. Decides what gets made.
Lead EngineerClaude — the AI. Designs, proposes, implements.

01WHAT IT IS

zmhstudio is a Claude Code plugin marketplace: a single repository that distributes shared studio tooling — as installable plugins — to every game project in the studio (an FPS, a JRPG, an action slicer, a Roblox experiment, and whatever comes next).

It exists to kill one specific failure mode: the copy-tailor-drift cycle. The same production commands used to be hand-copied into every game repo and re-tailored on the spot — so every time one repo learned something, the other copies quietly fell behind. Now there is exactly one implementation of each tool, and each game repo supplies only a thin config describing what makes it different (its build shell, its validation gate, its publish mechanics).

02THE PRODUCTION LOOP

Every project runs the same loop, driven by the zmh-producer plugin. The Creative Director's decisions enter at the top; working, validated features come out the bottom — with the AI doing every step in between autonomously.

BACKLOGevery idea lives in a bucketed TODO, groomed by interview
▶
PLANeach item becomes a written, reviewable implementation plan
▶
OVERSEEthe AI implements the plan end-to-end in an isolated worktree
▶
GATEnothing integrates without a green validation pass
▶
SHIPintegrate, publish, and fold learnings back into context

A ledger records every run — which model did the work, whether it passed first try, where it escalated — and a mining command periodically reads it to route future tasks to the cheapest model tier that can actually do them.

03THE PLUGIN FAMILY

Every plugin carries the zmh- prefix on purpose: typing /zmh in Claude Code autocompletes the whole family. Build order: producer → synth → 3d → pixart.

zmh-producer LIVE

The production loop itself. Seven commands that take a repo from empty scaffold to autonomous burndown:

  • studio-init — bootstrap a new game repo by interview
  • backlog-plan-gen / backlog-refine — backlog → reviewed plans
  • oversee-implementation — one plan, implemented end-to-end, gated
  • backlog-burndown — run the whole plan-ready queue, serially
  • refine-context / ledger-report — the studio learns from itself

zmh-3d LIVE

Four 3D asset pipelines, all CPU-only and deterministic:

  • voxel characters — seeded zombies (6 decay stages) & dog variants
  • procedural scenery — a full town kit: 13 buildings, 5 roof forms, 19 texture recipes, world-edge mountains & river
  • geo terrain — real map coordinates → playable terrain kit
  • photogrammetry scan — phone video/photos → textured GLB, with ML monocular depth and silhouette hull completion

zmh-pixart LIVE

Deterministic pixel-icon pipeline. A JSON icon spec — palette plus an ordered list of drawing ops (rects, lines, discs, mirrored bitmaps) — renders to crisp PNGs or a single vendorable TypeScript module. Same spec in, same pixels out, forever.

zmh-synth PARTIAL

Music & SFX synthesis. The knowledge layer is live: a tracker-style score format, synthesized-instrument recipes, and genre grids that already produced a shipped ten-track game soundtrack. The render-to-file CLI is next on the roadmap.

04HOUSE RULES

Deterministic by default Same spec + same seed = identical output, every time. Assets are regenerable artifacts, not precious binaries.
Stdlib-only cores The pipeline CLIs run on plain Python — no dependency trees to rot. MCP servers are thin adapters; the CLI is always the source of truth.
Generic spine, per-repo config Shared commands never special-case one game. Anything repo-specific lives in that repo's config file — the anti-drift rule the studio is built on.
Gates before merges Command edits pass LLM-judged evals; implementations pass validation gates; plans are preflighted against the real tree before any work starts.
Plans are permanent Every design doc stays in the repo and marks itself IMPLEMENTED when it lands — the "why" is never deleted.
Wholly-owned assets Seed templates are authored programmatically with a license ledger — no scraped models, no rights ambiguity.

06UNDER THE HOOD

Inside any repo with access, the whole studio installs in two lines of Claude Code:

> /plugin marketplace add ZackHelms/zmhstudio
> /plugin install zmh-producer@zmhstudio
# the marketplace repo is private — installs ride on existing git credentials

One repo, many plugins. Each game repo installs what it needs and adds a thin producer.md config; the marketplace never reaches into a game repo, and a game repo never forks the shared tooling. That boundary is the whole point.
